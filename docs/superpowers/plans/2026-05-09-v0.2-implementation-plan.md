# Mool v0.2 — Microphone Audio Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a microphone-on toggle to the Recorder page (default on) so screen recordings carry the user's narration. Mic permission is requested before screen-share at Start time; the mic audio track is merged into the screen MediaStream and recorded as one `video/webm;codecs=vp9,opus` blob.

**Architecture:** Extend the existing pure-reducer + adapter split in `app/src/public/recorderFlow.js` and `app/src/public/recorder.js`. New state (`RequestingMic`), two new events (`UserMediaGranted`, `UserMediaFailed`), one new effect (`requestUserMedia`), and one extended effect (`startRecording` gains an optional `audioStream` field). The `StartClicked` event gains a required `audioEnabled: boolean` payload that the adapter reads from the DOM checkbox at click time. Server, R2 layout, and SQLite schema are unchanged; only `ALLOWED_MIME` in `recording.ts` gains two new entries for the `,opus` codec strings.

**Tech Stack:** Node.js + Express + TypeScript, plain HTML + vanilla JS (ES modules, no bundler), `MediaRecorder` + `getUserMedia` + `getDisplayMedia` browser APIs, vitest + supertest for tests.

**Spec:** `docs/superpowers/specs/2026-05-09-v0.2-microphone-audio-design.md`

**Pre-flight:** Working directory must be the `feature/v0.2-microphone-audio` branch (already created from `main` and carrying the spec in `docs/superpowers/specs/`). Run `git status` first to confirm a clean working tree.

## Task ordering rationale

Tasks are ordered so each one is independently testable and committable:

1. **Server allow-list first** (Task 1). It's a pure addition with no behavior coupling — adds the new mime strings to `ALLOWED_MIME` and pins them with tests. Everything downstream depends on the server accepting these strings.
2. **State machine reducer next** (Tasks 2–4). Pure functions, fully unit-tested. The reducer must be correct before the adapter can wire it up.
3. **HTML + CSS** (Task 5). Trivial, but needed before the adapter can read the checkbox.
4. **Adapter** (Task 6). Glues the new SM transitions to `getUserMedia` and stream merging.
5. **Manual smoke checklist** (Task 7). The browser-side integration is verified by hand per §8 of the spec; no automated browser tests are added (consistent with the v0.1 boundary).

```javascript
async function requestUserMedia() {
  if (!navigator.mediaDevices || !navigator.mediaDevices.getUserMedia) {
    dispatch({
      type: 'UserMediaFailed',
      reason:
        'Microphone API unavailable. The page must be served over https or http://localhost — check your URL.',
    });
    return;
  }
  let stream;
  try {
    stream = await navigator.mediaDevices.getUserMedia({ audio: true });
  } catch (err) {
    const name = err && err.name ? err.name : 'Error';
    let message;
    if (name === 'NotFoundError') {
      message = 'No microphone found. Turn off the mic toggle to record silently.';
    } else if (name === 'NotAllowedError' || name === 'SecurityError') {
      message =
        'Microphone access denied. Allow it in your browser, or turn off the mic toggle to record silently.';
    } else {
      const detail = err && err.message ? err.message : String(err);
      message = `Microphone error: ${name} — ${detail}`;
    }
    dispatch({ type: 'UserMediaFailed', reason: message });
    return;
  }
  // Hold the stream in adapter state so releaseStream can clean it up
  // even if the screen-share prompt is denied (no startRecording yet).
  activeAudioStream = stream;
  dispatch({ type: 'UserMediaGranted', stream });
}
```

The reason strings here are the canonical text the spec calls for in §5; they reach the user via the `setStatus` effect in the `UserMediaFailed` reducer transition.

- [ ] **Step 4: Update `startRecording` effect to merge audio when present**

Find the existing `startRecording` function (lines 134-156). Replace its signature and body to accept and merge an optional audio stream. The new function:

```javascript
/**
 * @param {MediaStream} stream
 * @param {MediaStream | undefined} audioStream
 */
function startRecording(stream, audioStream) {
  activeStream = stream;
  activeAudioStream = audioStream ?? null;

  // Merge mic audio into the screen stream so a single MediaRecorder
  // produces one container with both video and audio tracks.
  if (audioStream) {
    for (const track of audioStream.getAudioTracks()) {
      stream.addTrack(track);
    }
  }

  chunks = [];
  const mimeType = pickMimeType(Boolean(audioStream));
  mediaRecorder = new MediaRecorder(stream, { mimeType });
  mediaRecorder.ondataavailable = (e) => {
    if (e.data && e.data.size > 0) chunks.push(e.data);
  };
  mediaRecorder.onstop = () => {
    const finalMime = mediaRecorder ? mediaRecorder.mimeType : mimeType;
    const blob = new Blob(chunks, { type: finalMime });
    dispatch({ type: 'RecorderStopped', blob, mimeType: finalMime });
  };
  // Either track ending (screen or mic) flows through Stopping.
  // TrackEnded outside Capturing is a no-op (see recorderFlow.js).
  for (const track of stream.getTracks()) {
    track.onended = () => dispatch({ type: 'TrackEnded' });
  }
  mediaRecorder.start();
}

/** @param {boolean} hasAudio */
function pickMimeType(hasAudio) {
  if (hasAudio) {
    if (MediaRecorder.isTypeSupported('video/webm;codecs=vp9,opus')) {
      return 'video/webm;codecs=vp9,opus';
    }
    if (MediaRecorder.isTypeSupported('video/webm;codecs=vp8,opus')) {
      return 'video/webm;codecs=vp8,opus';
    }
    return 'video/webm';
  }
  if (MediaRecorder.isTypeSupported('video/webm;codecs=vp9')) {
    return 'video/webm;codecs=vp9';
  }
  return 'video/webm';
}
```

And update the call site in `runEffect` (the `case 'startRecording':` arm):


This handles both cases: (a) `Starting{audioStream}` + `DisplayMediaFailed` (only `activeAudioStream` is set, since `startRecording` never ran — Step 3 assigns `activeAudioStream` at mic-acquisition time so releaseStream has something to release), and (b) the normal stop flow where both streams are alive (`activeStream` was assigned by `startRecording` and `activeAudioStream` was assigned by `requestUserMedia`).

The reducer never needs to know about adapter-side cleanup; it just emits `releaseStream` whenever a stream might be alive, and the adapter is responsible for the rest.
