# Mool v0.1 — Anonymous Screen Recorder

**Date:** 2026-05-09
**Status:** Approved design, pending implementation plan
**Project root:** `/home/toyin/mool`

## 1. Goal

Ship the irreducible Loom-like core loop: **record screen → get share link → viewer plays in browser**. Nothing else. The product is self-hosted on Toyin's Ubuntu server, exposed via Cloudflare Tunnel, with video files stored in Cloudflare R2.

This is v0.1 of a project intended to grow into a business. The wedge (target audience or feature edge) is intentionally undecided — v0.1 builds the universal core so that future versions can lean into whichever angle proves promising.

## 2. Scope

### In scope for v0.1

- A single web page with a "Start Recording" button
- Browser-based screen capture via `getDisplayMedia` and MediaRecorder
- Direct browser-to-R2 upload via presigned URL
- Slug generation and metadata storage in SQLite
- Public share link of the form `record.<domain>/v/<slug>`
- Viewer page that plays the recording in a `<video>` element streamed from R2
- Docker Compose deployment with two services: the Node app and `cloudflared`

### Explicitly out of scope for v0.1

- User accounts, authentication, login
- Microphone audio (added in v0.2)
- Webcam PIP overlay (added in v0.3)
- Recording library / "my recordings" view
- Editing, trimming, thumbnails, previews
- Comments, reactions, or analytics
- File expiry, lifecycle policies, or storage quotas
- Rate limiting beyond what Cloudflare provides at the edge
- Email, notifications, or sharing integrations

## 3. User flow

1. User visits `record.<domain>` over HTTPS (terminated at Cloudflare).
2. Recorder page loads; user clicks **Start Recording**.
3. Browser prompts for screen-share permission via `getDisplayMedia`.
4. MediaRecorder begins capturing the stream as `video/webm; codecs=vp9`. UI shows a stop button and elapsed timer.
5. User clicks **Stop**.
6. Browser POSTs to `/create-upload` with `{contentType, sizeBytes}`.
7. Server generates a 6-character base62 slug, mints a presigned R2 PUT URL, writes a row to SQLite, and returns `{slug, uploadUrl, viewerUrl}`.
8. Browser PUTs the recorded blob directly to R2 using the presigned URL. The home server is not in the upload byte path.
9. On successful PUT, the recorder page displays the share URL `record.<domain>/v/<slug>` with a copy button.
10. A viewer who opens the share URL receives a viewer page whose `<video src>` points to the public R2 custom-domain URL `videos.<domain>/<slug>.webm`. Playback bytes flow from R2 / Cloudflare CDN directly to the viewer.

## 4. Architecture

```
[ Browser ]
   │
   │ 1. Loads recorder/viewer HTML
   ▼
[ Cloudflare Tunnel → Node app on home server ]
   │
   │  Serves static HTML.
   │  POST /create-upload  → mints presigned R2 PUT, writes SQLite row,
   │                         returns { slug, uploadUrl, viewerUrl }
   │
   ▼
[ Browser PUTs file directly to R2 ]
   │
   ▼
[ Cloudflare R2 ] ──► public via custom domain videos.<domain>
                              │
                              ▼
                  Viewers stream from Cloudflare's edge
                  (home server is not in the playback byte path)
```

### Components

| Component         | Choice                                | Notes                                              |
| ----------------- | ------------------------------------- | -------------------------------------------------- |
| Container runtime | Docker Compose                        | Two services: `app`, `cloudflared`                 |
| Backend           | Node.js + Express + TypeScript        | Single process                                     |
| Frontend          | Plain HTML + vanilla JS               | Two pages: recorder, viewer. No build step.        |
| Metadata DB       | SQLite                                | One file in mounted `/data` volume                 |
| Object storage    | Cloudflare R2                         | S3-compatible; AWS SDK works unmodified            |
| Ingress           | Cloudflare Tunnel                     | No port forwarding, free TLS, edge for static HTML |
| Recording format  | `video/webm; codecs=vp9`              | Browser-native via MediaRecorder                   |
| AWS SDK           | `@aws-sdk/client-s3` + presigner v3   | Used to talk to R2                                 |

### Why this shape

- **Browser uploads directly to R2.** Removes home upload bandwidth as a bottleneck — the most common scaling problem for self-hosted media services.
- **Viewers stream directly from R2.** Removes home server from the playback path entirely. If the home server is down, existing recordings still play; only the ability to start *new* recordings is affected.
- **Home server is small.** It serves static HTML, mints presigned URLs, and reads/writes a SQLite file. No media bytes pass through it. Resource footprint is trivial.
- **Single SQLite file.** One file, no DB server process, no schema migrations infrastructure for v0.1. Easily swapped for Postgres at v0.4 when accounts arrive.

## 5. Data model

SQLite schema (`data/db.sqlite`):

```sql
CREATE TABLE recordings (
  slug        TEXT PRIMARY KEY,
  r2_key      TEXT NOT NULL,
  mime_type   TEXT NOT NULL,
  created_at  INTEGER NOT NULL  -- unix epoch milliseconds
);

CREATE INDEX idx_recordings_created_at ON recordings(created_at);
```

`size_bytes` is intentionally omitted in v0.1 — we don't need it for any user-facing functionality and it adds a moving part (which side is authoritative if browser-reported size differs from R2's actual size?). Add it later if needed.

## 6. API surface

| Method | Path             | Purpose                                                                        |
| ------ | ---------------- | ------------------------------------------------------------------------------ |
| GET    | `/`              | Recorder page (static HTML)                                                    |
| GET    | `/v/:slug`       | Viewer page (HTML with public R2 URL embedded)                                 |
| POST   | `/create-upload` | Generate slug, mint presigned R2 PUT URL, write SQLite row                     |

`POST /create-upload` request body:

```json
{ "contentType": "video/webm", "sizeBytes": 12345678 }
```

Server validates `sizeBytes <= MAX_UPLOAD_BYTES` before signing. The signed URL bakes `Content-Length: sizeBytes` into the request, so R2 rejects PUTs whose actual body length doesn't match the declared size.

Response:

```json
{
  "slug": "x7k2p9",
  "uploadUrl": "https://<account>.r2.cloudflarestorage.com/<bucket>/x7k2p9.webm?X-Amz-...",
  "viewerUrl": "https://record.<domain>/v/x7k2p9"
}
```

**R2 key layout:** v0.1 stores recordings flat at the bucket root as `<slug>.webm`. This keeps the public URL `videos.<domain>/<slug>.webm` a one-to-one map with the R2 key. A `recordings/` (or other) prefix can be introduced later when the bucket holds multiple content types (e.g. thumbnails in v0.5).

## 7. Slug generation

- Format: 6-character base62 (`[A-Za-z0-9]`), giving ~57 billion combinations.
- Generated server-side using a CSPRNG.
- On `INSERT INTO recordings`, rely on the `PRIMARY KEY` constraint; on conflict, retry with a fresh slug. With ~57 billion namespace and v0.1 traffic, collisions are negligible but the retry guards against it cleanly.

## 8. Security model

- **Public bucket, security through unguessable slugs.** Same model as Loom unlisted videos. The 6-char base62 namespace makes URL guessing infeasible at v0.1 scale.
- **No authentication in v0.1.** Anyone with the recorder URL can create a recording.
- **Cloudflare in front.** Provides free TLS, basic DDoS protection, and edge caching. Tunnel hides the home server's IP.
- **CORS on R2 bucket** restricted to the recorder origin only, so the presigned PUT endpoint cannot be exploited from arbitrary origins.

This security posture is acceptable for a personal / closed-beta v0.1. **Before opening to the public**, at minimum we will need: rate limiting on `/create-upload`, abuse reporting, and likely auth (which arrives in v0.4 per the growth ladder).

## 9. Deployment topology

`docker-compose.yml` lives at the project root with two services:

- **`app`** — built from `./app`, runs the Node/Express server on port 3000 (internal only). Mounts `./data` for SQLite. Reads R2 credentials and config from environment variables.
- **`cloudflared`** — official `cloudflare/cloudflared:latest` image, runs the Cloudflare Tunnel. Routes `record.<domain>` traffic to `app:3000` based on the tunnel configuration set in the Cloudflare dashboard. Reads `TUNNEL_TOKEN` from environment.

The home server only needs Docker + Docker Compose installed (already true). All other dependencies live inside containers.

### Environment variables (set via `.env`, not committed)

```
R2_ACCESS_KEY_ID=...
R2_SECRET_ACCESS_KEY=...
R2_BUCKET=mool-recordings
R2_ENDPOINT=https://<account-id>.r2.cloudflarestorage.com
R2_PUBLIC_BASE_URL=https://videos.<domain>
PUBLIC_APP_URL=https://record.<domain>
TUNNEL_TOKEN=...
PORT=3000
DATA_DIR=/data
MAX_UPLOAD_BYTES=524288000   # 500 MB cap, enforced when minting presigned PUT
```

### One-time Cloudflare setup (manual, outside this repo)

1. Create R2 bucket `mool-recordings`.
2. Create R2 API token scoped to that bucket with read+write.
3. Attach a custom domain to the bucket (e.g. `videos.<domain>`).
4. Add a CORS policy on the bucket allowing `PUT` from `https://record.<domain>` with `Content-Type` header.
5. Create a Cloudflare Tunnel routing `record.<domain>` → `http://app:3000`.
6. Save the tunnel token and R2 credentials into the server's `.env`.

## 10. Operational limits & defaults (v0.1)

- **Max upload size:** 500 MB. The browser reports the recording's `sizeBytes` in `POST /create-upload`. The server rejects requests over `MAX_UPLOAD_BYTES`, then signs `Content-Length: sizeBytes` into the presigned PUT URL. R2 rejects PUTs whose actual body length differs from the signed size, so the browser cannot exceed the declared size after the URL is issued.
- **No rate limiting** beyond Cloudflare's edge defaults.
- **No file expiry / cleanup.** Recordings persist indefinitely. A retention/lifecycle policy will be added before public launch.
- **No backups of SQLite.** v0.1 metadata loss is recoverable (we can rebuild a recordings index by listing the R2 bucket); this becomes more important with accounts in v0.4.

## 11. Risks & mitigations

| Risk                                              | Likelihood | Impact                                       | Mitigation in v0.1                                     |
| ------------------------------------------------- | ---------- | -------------------------------------------- | ------------------------------------------------------ |
| Home server downtime                              | Medium     | New recordings cannot start; existing ones still play | Accept for v0.1; existing-content availability is preserved by R2 |
| Abuse — unauthenticated upload endpoint           | Medium     | Storage cost + reputational                  | Slug-only access, R2 free tier headroom, will add rate limiting before public launch |
| R2 / Cloudflare service outage                    | Low        | Full outage during the window                | Accept; this is the price of using a managed CDN/storage |
| Browser-side recording quality limits             | Low        | Long recordings may have framedrops          | Document; revisit if it bites users                    |
| SQLite corruption                                 | Low        | Metadata loss, but R2 bucket is intact       | WAL mode + nightly backup of `data/db.sqlite` (defer to a follow-up if not needed in v0.1) |

## 12. Growth ladder

| Version | What's added                                       | Why                                                       |
| ------- | -------------------------------------------------- | --------------------------------------------------------- |
| v0.2    | Microphone audio                                   | Most useful recordings need narration                     |
| v0.3    | Webcam PIP overlay                                 | The "talking head" Loom signature                         |

Each version should ship behind the same architectural shape established here. The two structural changes on the ladder are: (a) SQLite → Postgres at v0.4, and (b) public bucket → presigned-GET access at v0.4 once auth is in place.
