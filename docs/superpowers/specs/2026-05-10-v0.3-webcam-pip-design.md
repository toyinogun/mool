# Mool v0.3 тАФ Webcam Picture-in-Picture

**Date:** 2026-05-10
**Status:** Approved design, pending implementation plan
**Project root:** `/home/toyin/mool`
**Builds on:** [`2026-05-09-v0.2-microphone-audio-design.md`](./2026-05-09-v0.2-microphone-audio-design.md)

## 1. Goal

Recordings carry a webcam picture-in-picture (bottom-left, circular) alongside the screen, so a viewer sees the speaker's face while watching the screen content. The Loom "screen + face" loop, with no other change in product surface area.

## 2. Scope

### In scope for v0.3

- A "ЁЯУ╖ Camera on / off" toggle on the Recorder page, sitting next to the existing mic toggle. Default OFF.
- Live preview circle on the recorder page when the toggle is on; mirrored locally; dismissible via a small "Hide preview" link without disabling the camera.
- Camera permission requested the moment the toggle flips on (so the preview can render). The camera stream is held by the adapter for the lifetime of the toggle-on state and released on toggle-off.
- Composite recording: a hidden canvas at the screen's native resolution; a `requestAnimationFrame` loop draws the screen frame, then the webcam clipped to a circle in the bottom-left at 18% of canvas height with a 4% margin. The webcam in the recording is **un-mirrored**.
- One MediaRecorder, one webm out, single presigned PUT to R2. R2 key format and viewer unchanged.
- Failure handling: camera permission denied at toggle-on, no camera device, camera disconnect mid-recording, screen-share denial when camera is already held.
- Tests covering the new state-machine threading and a manual-smoke matrix for the visual paths.

### Explicitly out of scope for v0.3

- Camera device picker (browser default device only, mirroring v0.2's mic decision).
- Configurable PIP position, size, or shape (one corner, one size, one shape).
- Camera-only mode (recording without a screen source).
- Background blur, virtual backgrounds, or face-tracking PIP.
- Mid-recording camera mute / hide (toggle while recording is in flight).
- Resolution constraints on `getUserMedia({video})` тАФ accept browser defaults.
- Any change to the server, the API surface, the SQLite schema, or the R2 key layout.

## 3. UI

The recorder page gains a second toggle pill on the same row as the mic toggle:

```
[ ЁЯОд Mic on тЬУ ]   [ ЁЯУ╖ Camera on тЦв ]
[  preview circle (when camera on, mirrored)  ]
[  Hide preview / Show  ]                   тЖР link beside the preview
[  Start Recording  ]   [ Stop ]
```

Both toggles share the same enabled/disabled lifecycle: enabled in `Idle | Done | Failed`; disabled and dimmed during all in-flight states.

When the camera toggle is on:
- A 110px CSS-mirrored preview circle renders below the toggle row, fed by the held camera `MediaStream`.
- Beside it, a "Hide preview" link collapses the circle to a placeholder strip ("Camera on, preview hidden тАФ Show"). Clicking "Show" restores the circle. The camera stream is **not** affected by hide/show.

Toggle state is owned by the DOM (the checkbox remembers its own checked state). The state machine reads both `audioEnabled` and `videoEnabled` from the `StartClicked` event payload at click time; it does not persist toggle state in the reducer.

## 4. User flow

### Toggle flow (decoupled from recording, owned by the adapter)

1. Toggle off тЖТ user clicks тЖТ browser shows camera permission prompt.
   - **Granted:** adapter holds the camera stream; preview circle renders below the toggle row.
   - **Denied / no device:** toggle bounces back to off; inline message ("Camera access denied. Allow it in your browser, or leave the camera off." or "No camera found. Recording will be screen-only.").
2. Toggle on, preview visible тЖТ click "Hide preview" тЖТ preview hidden, camera still held.
3. Toggle on тЖТ click toggle тЖТ adapter releases the camera stream, preview removed, toggle off.

### Recording flow with camera on

1. User clicks Start.
2. If mic toggle is on, browser prompts mic (v0.2 path, unchanged). If off, skip.
3. Browser prompts screen-share.
4. Camera stream is **already held** from the toggle flow тАФ no extra permission prompt.
5. Adapter constructs the composite: hidden canvas at the screen stream's native resolution, rAF loop drawing screen frame + clipped-circle webcam in the bottom-left. `canvas.captureStream(30)` produces the video track; mic audio (if present) is added as a track on the same MediaStream. A single MediaRecorder consumes the composite stream.
6. Stop / upload / share-link flow is identical to v0.2.

### Recording flow with camera off

Identical to v0.2. Canvas/rAF/composite path is bypassed тАФ `MediaRecorder` consumes the raw screen+mic stream.

## 5. Failure modes

| Failure | Where | What user sees | State after |
| --- | --- | --- | --- |
| Camera permission denied | Toggle flip on | "Camera access deniedтАж" inline; toggle bounces to off | `Idle`; camera off |
| No camera device (`NotFoundError`) | Toggle flip on | "No camera foundтАж" inline; toggle bounces to off | `Idle`; camera off |
| Mic permission denied (v0.2 unchanged) | Step 2, mic toggle on | v0.2 message | `Failed`; camera stream **stays held** (toggle still on) |
| Screen denied after mic+cam held | Step 3 | v0.1 screen-share message; mic stream released, **camera stream stays held** | `Failed`; camera on |
| Camera track ends mid-recording (e.g. USB cam unplugged) | During `Capturing` | Status: "Camera disconnected тАФ continuing with screen only." Composite switches to drawing only the screen frame. | `Capturing` continues; recording becomes screen-only from that point |
| Screen track ends mid-recording (v0.1 behavior) | During `Capturing` | v0.1 message | `Stopping` then upload |

### Two deliberate divergences from v0.2

**1. Camera mid-recording disconnect = "continue without camera," not "stop and upload."**
v0.2's mic-track-end rule is "stop and upload" because mic is integrated into the only audio track. v0.3's camera-track-end rule is "continue without camera" because the screen content is the primary signal in a screen recording тАФ killing a 20-minute recording because the USB webcam fell out would be infuriating. The composite module handles `cameraTrack.onended` by flipping an internal flag; the next rAF tick draws screen only. MediaRecorder is unaffected. The output is one webm where the PIP disappears partway through.

**2. Camera stream survives recording-flow failures.**
Because the camera lives outside the recording state machine (held by the toggle, not by Start), screen-share failure after mic+cam are held releases the mic but **not** the camera. The toggle is still on; the user can retry without re-authorising the camera.

## 6. State machine extension

The v0.1 SM already separates pure reducer (`recorderFlow.js`) from adapter (`recorder.js`). v0.2 widened payloads to carry the mic stream. v0.3 widens payloads further to carry the camera-enabled flag, and **does not introduce any new state variants, transitions, or effects**.

### Architectural call: camera lives in the adapter, not the reducer

v0.2 acquired the mic at Start time via a `RequestingMic` reducer state because mic has no preview need. v0.3's camera is acquired at *toggle time* so the preview can render тАФ there is no recording-flow state to model. The adapter owns the camera stream's lifecycle (toggle handler in/out); the reducer never holds a `MediaStream` for video.

This keeps the reducer focused on "the recording flow" and avoids a `RequestingCamera` state that would have no useful in-flight semantics: the stream is already held by the time `StartClicked` fires.

### Modified event

- `StartClicked { audioEnabled: boolean, videoEnabled: boolean }` тАФ the adapter reads both DOM toggles at click time and embeds both flags. The reducer threads `videoEnabled` through state until `startRecording` is emitted. The reducer never branches on `videoEnabled` for state selection тАФ it only carries it forward.

### Modified states (payload widening only тАФ no new variants)

- `RequestingMic { videoEnabled }` тАФ carries the flag from `StartClicked` until `UserMediaGranted` arrives.
- `Starting { videoEnabled }` тАФ toggle-off mic case, with the flag.
- `Starting { audioStream, videoEnabled }` тАФ toggle-on mic case, with the flag.

### Modified effect

- `startRecording { stream, audioStream?, videoEnabled }` тАФ adds `videoEnabled`. The adapter, on receiving this effect with `videoEnabled === true`, looks up its own held camera stream and routes through the composite module before constructing the MediaRecorder. With `videoEnabled === false`, behaviour is identical to v0.2.

### No new states, transitions, effects, or mime types

The v0.2 transition graph (`Idle | Done | Failed тЖТ тАж тЖТ Capturing тЖТ Stopping тЖТ MintingUrl тЖТ Uploading тЖТ Done`) is unchanged. No new `Requesting*` state for the camera. No new failure transitions in the reducer for camera errors (those are handled outside the SM, in the toggle handler).

## 7. Compositing details

A new module: `app/src/public/recorderComposite.js`.

### Module surface

```
composeStreams({ screenStream, cameraStream, audioStream })
  тЖТ { compositeStream, stop, onCameraEnded(cb) }
```

- Sets up a hidden `<canvas>` at the screen stream's native resolution (`screenStream.getVideoTracks()[0].getSettings().{width,height}`).
- Creates two off-screen `<video>` elements, one bound to the screen track, one to the camera track.
- Runs a `requestAnimationFrame` loop that, each tick, draws the screen frame to the canvas, then (if camera is still active) clips a circle in the bottom-left and draws the webcam frame inside it.
- Produces the composite video track via `canvas.captureStream(30)`. If `audioStream` is present, its audio track is `addTrack`-ed onto the same MediaStream so a single MediaRecorder muxes both.
- Listens for `cameraTrack.onended`; on fire, flips an internal flag so subsequent rAF ticks skip the camera draw, and invokes the `onCameraEnded` callback (the adapter uses this to update the status text).
- `stop()` cancels the rAF loop, removes the off-screen video elements, and stops the tracks the module owns (the composite video track produced by `captureStream`). It does **not** stop the input screen/camera/audio tracks тАФ those are owned by the adapter, which decides when to release them.

### Geometry

- **Canvas resolution:** matches the screen track's `getSettings().{width,height}`. No fixed canvas size.
- **PIP shape:** circle (canvas `arc()` clip path before drawing the webcam frame).
- **PIP diameter:** `0.18 ├Ч canvasHeight`. Example: 194px on 1080p.
- **PIP position:** bottom-left, `0.04 ├Ч canvasWidth` from left and `0.04 ├Ч canvasHeight` from bottom.
- **Webcam frame:** center-cropped to a square before draw, so 16:9 or 4:3 webcam feeds aren't distorted by the circular clip.
- **Border:** none in v1. No drop shadow. Minimal.

### Mirror

- **Preview** (`<video>` on the recorder page): mirrored via CSS `transform: scaleX(-1)`. The mirror is purely visual; the underlying stream is un-mirrored.
- **Recording**: drawn un-mirrored. So on-screen text and gestures read correctly to viewers.

### Frame rate

- `canvas.captureStream(30)` produces a 30fps video track.
- The rAF callback is browser-paced (~60fps on most hardware). The loop draws on every tick; `captureStream` samples the canvas at its own 30fps cadence regardless of how often the canvas is updated, so over-drawing is cheap and under-drawing (e.g., main-thread stalls) just yields fewer captured frames.

### Encoding

Mime-type negotiation is unchanged from v0.2. The composite stream feeds the same MediaRecorder; codec choice is independent of source.

- With audio: `video/webm;codecs=vp9,opus` тЖТ `video/webm;codecs=vp8,opus` тЖТ `video/webm`.
- Without audio: `video/webm;codecs=vp9` тЖТ `video/webm`.

`ALLOWED_MIME` is unchanged. R2 key format unchanged. Viewer unchanged. Schema unchanged.

## 8. Adapter wiring

### Camera lifecycle (toggle-driven)

The adapter (`recorder.js`) gains:

- A camera-toggle click handler. Off тЖТ on: call `requestCamera()` (`navigator.mediaDevices.getUserMedia({ video: true, audio: false })`), hold the stream, render the preview circle. On тЖТ off: stop the stream's tracks, remove the preview, set the toggle visually to off.
- A "Hide preview" / "Show" handler that toggles the preview element's visibility only. Does not touch the stream.
- An error path on `requestCamera()` rejection: bounce the toggle back to off, render the failure message inline, do not affect any other state.

### Composite wiring (Start-driven)

In the `startRecording` effect handler:

- If `videoEnabled === true`: call `composeStreams({ screenStream, cameraStream: this.cameraStream, audioStream })`, register the `onCameraEnded` callback (updates status text), pass `compositeStream` to `recorderCapture.js`.
- If `videoEnabled === false`: pass the raw screen+mic stream as v0.2 does.

When the recording stops (Stop click or screen-track-end), the adapter calls the composite's `stop()` so the rAF loop is cancelled. The adapter does **not** stop the camera stream here тАФ that's the toggle-off handler's job.

### `recorderCapture.js` is not taught about cameras

`recorderCapture.js` continues to take "the stream to record" and run the `MediaRecorder` lifecycle. Whether that stream is a raw `getDisplayMedia` stream or a canvas-composited stream is invisible to it. Approach **B** from brainstorming: the adapter pre-composites; the capture module stays focused on `MediaRecorder`.

## 11. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| Canvas compositing too slow тЖТ dropped frames in recording | Medium | Choppy PIP video | 30fps target; draw every rAF tick. If real-user perf complaints, defer to OffscreenCanvas/Worker in v0.3.5 |
| Webcam aspect ratio mismatch in circle | Low | Squashed face | Center-crop to square before draw |
| Mirror confusion (preview vs recording) | Low | "Why is my text reversed?" | Caption "Mirrored preview тАФ viewers see un-mirrored" beside preview |
| Camera permission previously denied | Medium | Toggle stuck off until browser settings fixed | Inline message names the missing permission; toggle bounces off |
| A/V drift on long recordings | Low | Out-of-sync output | One MediaRecorder muxes a single composite + audio; browser timestamps both |
| rAF loop leak after recording stops | Low | Page slows over time | `composeStreams()` returns `stop()` that cancels rAF and releases the canvas |
| Camera held forever after error path | Low | Tab camera indicator stays on | Toggle-off explicitly releases; recording-failure paths leave it on intentionally (user controls release) |

## 12. Out of scope, deferred to later versions

- **Camera device picker.** Defer until accounts (v0.4) make per-user device preferences worth persisting.
- **Configurable PIP position, size, or shape.** One corner, one size, one shape in v1; revisit on real-user feedback.
- **Camera-only mode** (no screen). Different flow shape (no `getDisplayMedia` call); deferred.
- **Background blur / virtual backgrounds.** Third-party libraries, not core.
- **Face-tracking PIP** that follows the speaker. Fancy, not core.
- **Mid-recording camera mute / hide** (live toggle while recording). Add only if a real user asks.
- **OffscreenCanvas / Worker compositing.** Performance optimisation; defer until needed.
- **Resolution / frame-rate constraints on `getUserMedia`.** Browser defaults accepted; revisit on a complaint.
