# Mool v0.2 — Microphone Audio

**Date:** 2026-05-09
**Status:** Approved design, pending implementation plan
**Project root:** `/home/toyin/mool`
**Builds on:** [`2026-05-09-v0.1-anonymous-recorder-design.md`](./2026-05-09-v0.1-anonymous-recorder-design.md)

## 1. Goal

Recordings carry the user's microphone audio alongside the screen video, so a viewer hears narration synced to what they see. The Loom "voice-over your screen" loop, with no other change in product surface area.

## 2. Scope

### In scope for v0.2

- A "🎤 Mic on / off" toggle on the Recorder page (default on), evaluated when the user clicks Start.
- Mic permission requested at Start time, before the screen-share prompt.
- Mic audio added as a track to the screen MediaStream and recorded as one `video/webm; codecs=vp9, opus` blob.
- Failure handling for mic permission denial, no mic device, and screen denial after mic was granted.
- Tests covering the new states and the audio path through the state machine.

### Explicitly out of scope for v0.2

- System / tab audio capture.
- Mid-recording mute, pause, or resume controls.
- Mic device selection UI (browser default device).
- Mic level meter or "on-air" visual indicator beyond the existing status text.
- Echo cancellation / noise suppression configuration UI (browser defaults accepted).
- Any change to the server, the API surface, the SQLite schema, or the R2 key layout.

## 3. UI

A single new control on the Recorder page, above the Start button:

```
[ 🎤 Mic on  ▢ ]   ← checkbox-style toggle, default checked
[  Start Recording  ]   [ Stop ]
```

The toggle is enabled only in `Idle`, `Done`, and `Failed` states (when starting a fresh capture is allowed). It is disabled and visually dimmed during all in-flight states (`Starting`, `RequestingMic`, `Capturing`, `Stopping`, `MintingUrl`, `Uploading`) so users cannot change it mid-flow.

Toggle state is owned by the DOM (a checkbox already remembers its own checked state). The state machine reads it at decision time via the `StartClicked` event payload; it does not persist toggle state in the reducer.

## 4. User flow

### Toggle ON (the new default)

1. User clicks Start.
2. Browser shows the mic permission prompt. User grants → mic MediaStream acquired.
3. Browser shows the screen-share prompt. User picks a screen → screen MediaStream acquired.
4. Adapter merges the mic audio track into the screen stream and starts a single `MediaRecorder`. Status reads `Recording…`, timer ticks.
5. User clicks Stop (or browser's "Stop sharing"). Upload proceeds exactly as v0.1. Result link is displayed.

### Toggle OFF

Identical to v0.1. No mic request. Recording is silent.

## 5. Failure modes

| Failure | Where | What user sees | State after |
| --- | --- | --- | --- |
| Mic permission denied | Step 2, toggle on | "Microphone access denied. Allow it in your browser, or turn off the mic toggle to record silently." | `Failed`; toggle stays on |
| No mic device available (`NotFoundError`) | Step 2 | "No microphone found. Turn off the mic toggle to record silently." | `Failed`; toggle stays on |
| User denies screen share after granting mic | Step 3 | v0.1 screen-share failure message; mic stream is released cleanly so the tab indicator goes off | `Failed`; toggle stays on |
| Mic track ends mid-recording (e.g. USB headphones unplugged) | During `Capturing` | Same as v0.1 `TrackEnded`: stop and upload whatever was captured | `Stopping` then onward |
| Screen track ends mid-recording (v0.1 behavior, unchanged) | During `Capturing` | v0.1 message | `Stopping` then onward |

The "stop & upload" choice on mid-recording mic-track-end mirrors v0.1's behavior for the screen track: do not lose the recording so far.

## 6. State machine extension

The v0.1 SM already separates pure reducer (`recorderFlow.js`) from adapter (`recorder.js`). v0.2 extends both without disturbing existing transitions when the toggle is off.

### New state

- `RequestingMic` — entered from `Starting` when the mic toggle is on; awaiting the user's response to the mic permission prompt. Holds no payload (the mic stream is captured by the adapter and handed back via `UserMediaGranted`).

### Modified state

- `Starting` becomes one of two shapes:
  - `{ kind: 'Starting' }` — toggle was off (v0.1 behavior).
  - `{ kind: 'Starting', audioStream: MediaStream }` — toggle was on, mic granted, awaiting screen.

### New events

- `UserMediaGranted { stream: MediaStream }` — adapter emits after `getUserMedia({audio:true})` resolves.
- `UserMediaFailed { reason: string }` — adapter emits on permission denial or `NotFoundError`. The reason string carries enough information for the failure-mode messages in §5.

### Modified event

- `StartClicked { audioEnabled: boolean }` — the adapter reads the toggle's checked value when the click fires and embeds it in the event. The reducer branches on this field.

### New effects

- `requestUserMedia` — adapter calls `navigator.mediaDevices.getUserMedia({audio:true})`, dispatches `UserMediaGranted` or `UserMediaFailed`.
- `startRecording` (existing) gains one optional field: `audioStream?: MediaStream`. When present, the adapter merges its audio track into the screen stream before constructing the MediaRecorder. When absent, behavior is identical to v0.1.
- `releaseStream` (existing) is extended in the adapter to also release the mic stream if one is held. It already releases the screen stream's tracks; the loop now covers both stored stream references.

### New transitions

Both branches of `StartClicked` also emit the v0.1 auxiliary effects `hideResult` and `setStatus{''}` to clear any stale share link or message, and `setButtons{startEnabled:false, stopEnabled:false}` to lock the buttons through the in-flight phase:

- `Idle | Done | Failed` + `StartClicked{audioEnabled:true}` → `RequestingMic` + `hideResult` + `setStatus{''}` + `setButtons{false,false}` + `requestUserMedia`.
- `Idle | Done | Failed` + `StartClicked{audioEnabled:false}` → `Starting` + `hideResult` + `setStatus{''}` + `setButtons{false,false}` + `requestDisplayMedia` (v0.1 path; regression-pinned).
- `RequestingMic` + `UserMediaGranted{stream}` → `Starting{audioStream:stream}` + `requestDisplayMedia`.
- `RequestingMic` + `UserMediaFailed{reason}` → `Failed{message:reason}` + `setStatus` + `setButtons{true,false}`.
- `Starting{audioStream}` + `DisplayMediaGranted{stream}` → `Capturing` + `startRecording{stream, audioStream}` + `setStatus{'Recording…'}` + `setButtons{false,true}` + `startTimer`.
- `Starting{audioStream}` + `DisplayMediaFailed` → `Failed` + `releaseStream` + `setStatus` + `setButtons{true,false}`.

The existing v0.1 `Starting{}` (toggle off) + `DisplayMediaFailed` transition is preserved unchanged: it does **not** emit `releaseStream`, since no stream has been acquired at that point. Only the new `Starting{audioStream}` variant emits `releaseStream` on display-media failure.

Out-of-state events in `RequestingMic` are no-ops, consistent with v0.1's "stale event = no-op" policy.

### Toggle UI state

The toggle's enabled/disabled state is managed by the adapter using the same pattern v0.1 uses for the Start/Stop buttons: the existing `setButtons` effect is extended (or paralleled by a new `setMicToggle{enabled}` effect — the implementation plan settles which) so every state transition that locks the buttons also locks the toggle, and every transition that unlocks them unlocks the toggle. The reducer does not own the toggle's checked value, only its enabled-ness.

### Toggle plumbing decision

The `audioEnabled` flag rides on the `StartClicked` event payload (the adapter reads the DOM checkbox at click time), rather than being persisted in reducer state via a separate `ToggleAudio` event. This keeps the reducer's view of the world to "what's currently happening in the recording flow" and treats the toggle as input at decision time. The reducer never needs to *cause* the toggle to change.

## 7. Encoding & format

### MediaRecorder mime-type negotiation

When `audioStream` is present, the adapter picks the first supported entry from this ordered list:

1. `video/webm;codecs=vp9,opus` — preferred.
2. `video/webm;codecs=vp8,opus` — fallback for browsers that don't expose VP9 in MediaRecorder.
3. `video/webm` — last resort; browser picks both codecs.

When `audioStream` is absent (toggle off), the v0.1 negotiation order is unchanged: `video/webm;codecs=vp9` → `video/webm`.

### Server-side mime-type allow-list

`POST /create-upload` validates the `contentType` against `ALLOWED_MIME` exported from `app/src/recording.ts` (per ADR-0006: the allow-list is a Recording-domain rule, not a wire shape, so it lives in the Recording module rather than a central contracts file). v0.2 adds two entries: `video/webm;codecs=vp9,opus` and `video/webm;codecs=vp8,opus`, and the `AllowedMime` union type alongside it gets the same two new members. `tests/contracts.test.ts` (which already pins the exact `ALLOWED_MIME` array) is updated to expect the new four-element list.

### R2 storage

The presigned PUT signs the content type into the URL; R2 stores it as the object's `Content-Type` header so the Viewer page's `<video>` tag plays it back correctly. The R2 key format is unchanged — still `<slug>.webm`. The codec parameter is metadata, not part of the path.

### Viewer

The `<video src="...">` element has no awareness of codecs. As long as the browser supports the codec combo at playback time, it just works. WebM/VP9/Opus has been universally supported in evergreen browsers for years. No viewer-side change in v0.2.

### Audio processing

`getUserMedia({audio:true})` applies the browser's default echo cancellation, noise suppression, and auto gain control. v0.2 accepts these defaults — they are well-tuned for voice-over and exposing knobs would be wasted UI surface. If users start complaining (e.g. "my mic sounds robotic"), we revisit by accepting `{audio: {echoCancellation, noiseSuppression, autoGainControl}}` from a settings panel in a later version.

## 8. Testing strategy

The v0.1 test split is preserved: pure reducer tests in `tests/recorderFlow.test.ts` (no DOM, no fetch, no MediaRecorder), HTTP-layer tests in `tests/recorderUpload.test.ts` against a fake fetch, route/server tests against supertest, contract tests pinning the wire shape.

### New / extended tests

1. **`recorderFlow.test.ts` — new state-machine cases:**
   - `StartClicked{audioEnabled:true}` from Idle → `RequestingMic` + `requestUserMedia`.
   - `StartClicked{audioEnabled:false}` from Idle → `Starting` + `requestDisplayMedia` (v0.1 path, regression-pinned).
   - `UserMediaGranted{stream}` from `RequestingMic` → `Starting{audioStream}` + `requestDisplayMedia`.
   - `UserMediaFailed{reason}` from `RequestingMic` → `Failed` with reason in message.
   - `DisplayMediaGranted{stream}` from `Starting{audioStream}` → `Capturing` + `startRecording{stream, audioStream}`.
   - `DisplayMediaFailed` from `Starting{audioStream}` → `Failed` + `releaseStream` (the mic-only release case).
   - Out-of-state events in `RequestingMic` are no-ops.

2. **`recorderFlow.test.ts` — exhaustiveness:** existing exhaustiveness tests get the new state, events, and effects added.

3. **`contracts.test.ts` — new allowed mime types:** cases for `video/webm;codecs=vp9,opus` and `video/webm;codecs=vp8,opus` accepted by `ALLOWED_MIME`; non-allowed audio codec strings still rejected.

4. **`createUpload.test.ts` — round-trip with new mime types:** a request with `contentType: 'video/webm;codecs=vp9,opus'` returns a presigned URL whose signed `Content-Type` matches.

5. **No new browser/integration tests.** v0.1 deliberately stayed off jsdom/Playwright; v0.2 holds that line. The mic acquisition itself (the actual `getUserMedia` call) lives in the adapter and is covered by manual smoke testing, the same way `getDisplayMedia` is in v0.1.

### Manual smoke after implementation

- Mic on, grant both prompts → recording plays back with audible voice in the Viewer.
- Mic on, deny mic → "Microphone access denied…" message; toggle stays on; clicking Start re-prompts.
- Mic on, grant mic, deny screen → mic indicator releases (verify in browser tab); failure message shown.
- Mic off → recording is silent (regression check on v0.1 behavior).
- Recording of ~5 minutes with mic on → audio stays synced with video on playback.

## 9. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| Audio/video drift on long recordings | Low | Voice-over slips out of sync | Both tracks share one MediaRecorder, which the browser muxes and timestamps together. Manual smoke covers ~5 min recording. |
| Browser doesn't expose Opus in MediaRecorder | Very low | No audio in recording despite toggle on | Negotiation falls back to `video/webm` (browser picks codecs); on the rare browser that won't combine, manual smoke catches it. |
| User confused by two consecutive permission prompts | Low | First-recording friction | Status text after first prompt explains what's happening; failure messages name the missing permission specifically. |
| Mic stream leak if screen denied | Low | Tab mic indicator stays on after failure | Explicit `releaseStream` effect on the `Starting{audioStream}` + `DisplayMediaFailed` transition (covered by test). |

## 10. Out of scope, deferred to later versions

- **System / tab audio.** May ship as a follow-up if a real user asks; the platform support cliff (Chrome tab-only, Firefox/Safari weak) makes it a poor v0.2 candidate.
- **Mid-recording mute / pause / resume.** Add as v0.2.5 if a real user asks.
- **Mic device picker.** Adds settings UI surface for a low-frequency need; defer until accounts (v0.4) make per-user preferences worth persisting.
- **Audio processing knobs.** Revisit only on a real complaint.
