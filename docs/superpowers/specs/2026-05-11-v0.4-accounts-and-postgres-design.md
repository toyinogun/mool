# Mool v0.4 — Accounts, Postgres, and Private Bytes

**Date:** 2026-05-11
**Status:** Approved design, pending implementation plan
**Project root:** `/home/toyin/mool`

## 1. Goal

Add accounts (magic-link sign-in via Resend), swap SQLite for Postgres, and move recording bytes behind server-minted signed-GET URLs. Signed-in users get a library of their own recordings; share links remain public-by-link — anyone with the link can view, no login required.

This is the v0.4 rung from the v0.1 spec's growth ladder, executed as a single coherent change. Accounts, the Postgres swap-in, and the public-bucket → presigned-GET access shift are bundled because they are mutually dependent: presigned-GET access needs auth (to know who created what), and ownership-aware schema is cleanest expressed in Postgres from the start.

## 2. Scope

### In scope for v0.4

- Magic-link sign-in (email-only, no passwords), sessions, sign-out
- Account model with display name (auto-derived from email local-part at first sign-in, edit flow deferred)
- Postgres replaces SQLite as the only metadata store, as a third service in `docker-compose.yml`
- Drizzle for schema definition and migrations
- Recorder page requires sign-in; `POST /create-upload` requires session
- `recordings` table gains a non-null `user_id` foreign key
- Viewer page mints a fresh, short-lived signed-GET URL on each page load; share links remain `record.<domain>/v/<slug>` and continue to work for anyone
- R2 bucket flips from public custom domain to private; the `videos.<domain>` binding is removed
- Library page (`/library`) — list of caller's recordings, delete action
- `GET /healthz` for readiness
- Fresh-start cutover: existing v0.1 SQLite is archived but not migrated; the R2 bucket is wiped

### Explicitly out of scope for v0.4

- Per-recording sharing settings (private / specific people)
- Display-name edit UI, account-settings page, avatars, profile pages
- Password fallback or OAuth providers
- Email change flow, account deletion (the schema is ready via `ON DELETE CASCADE`, but no UI)
- Rate limiting on `POST /auth/request-link` beyond Cloudflare's edge defaults
- Recording quotas / storage caps
- Thumbnails, titles, view counts (those are v0.5+ per the growth ladder)
- Auto-refresh on signed-GET 403 (deferred to v0.5 polish)
- Migration of v0.1 SQLite rows into Postgres

## 3. User flows

### Sign in (also the sign-up path — first magic-link click creates the account)

1. Signed-out user visits `record.<domain>/` → server 302s to `/signin`.
2. User enters email → `POST /auth/request-link {email}` → server normalizes (`trim().toLowerCase()`), runs a basic format regex; on format failure returns `204` without sending. On valid format: deletes any existing unconsumed `signin_tokens` rows for this email, generates a 32-byte token, stores `sha256(token)` with a 15-minute expiry, sends a Resend email containing `${PUBLIC_APP_URL}/auth/callback?token=<raw>`, returns `204`. The response is `204` for both cases so the client cannot distinguish "valid email, sent" from "invalid format, ignored".
3. User clicks the link → `GET /auth/callback?token=...` → server hashes, looks up, verifies not expired and not consumed, marks `consumed_at = NOW()`, `INSERT ... ON CONFLICT` on the `users` table (creates row on first sign-in, with `display_name = email local-part`), creates a session row, sets the `mool_session` cookie, 302s to `/`.

### Sign out

`POST /auth/signout` → server deletes the session row, clears the `mool_session` cookie, 302s to `/signin`.

### Record

1. Signed-in user lands on `/` — same recorder UI as v0.3, with a header strip showing display name and a "Library" link.
2. User clicks **Stop** → `POST /create-upload` (session-authenticated) → server generates slug, mints presigned PUT URL, `INSERT INTO recordings (slug, user_id, r2_key, mime_type)` with `user_id = req.user.id`, returns `{slug, uploadUrl, viewerUrl}`.
3. Browser PUTs the recorded blob directly to R2. Home server is not in the upload byte path.
4. Recorder page shows the share URL with a copy button, plus a "Saved to your library →" link.

### View (anyone with the link, signed in or not)

1. Viewer hits `record.<domain>/v/:slug`.
2. Server looks up the slug. 404 if not found. **No ownership check** — share links are public-by-link.
3. Server mints a fresh signed-GET URL against `r2_key` with a 1-hour TTL.
4. Server renders the viewer HTML with the signed URL inlined into `<video src="...">`.
5. Browser fetches bytes directly from R2 / Cloudflare's edge. Home server is not in the playback byte path.
6. If the viewer leaves the page open and starts playback after >1h, the next byte fetch 403s and HTML5 video errors. Recovery: a full page reload reissues a fresh URL. Auto-refresh on 403 is deferred to v0.5.

### Library (`/library`, signed in only)

1. Lists the caller's own recordings, newest first, paginated 50 per page via `?before=<created_at>` cursor.
2. Each row shows: thumbnail placeholder, created date, slug, copy-link button, delete button.
3. **Delete** → confirm modal → `DELETE /recordings/:slug` → server verifies `recordings.user_id = req.user.id`, deletes the R2 object, deletes the Postgres row.

## 4. Architecture

The byte-path shape is unchanged from v0.1: browser uploads directly to R2, viewers stream directly from R2. The home server still only serves HTML, mints presigned URLs, and reads/writes metadata — now to Postgres instead of SQLite.

```
[ Browser ]
   │
   │ Loads /signin, /, /v/:slug, /library
   ▼
[ Cloudflare Tunnel → Node app on home server ]
   │
   │  - Serves recorder/viewer/signin/library HTML
   │  - POST /auth/request-link → Resend → email to user
   │  - GET  /auth/callback     → upsert user, mint session cookie
   │  - POST /create-upload     → presigned PUT, INSERT recordings (auth req'd)
   │  - GET  /v/:slug           → mint signed-GET, render viewer HTML
   │  - DELETE /recordings/:slug→ R2 DELETE + Postgres DELETE
   │
   │              ┌─────────────┐
   ├──────────────►  Postgres   │  (sibling compose service)
   │              └─────────────┘
   ▼
[ Browser PUTs file directly to R2 ]
   │
   ▼
[ Cloudflare R2 ]  (private bucket — no public custom domain)
   ▲
   │ signed-GET URLs (1h TTL) embedded in viewer page <video src>
   │
   └──────  Viewers stream bytes from Cloudflare's edge
```

### What changes from v0.1

- `app` service depends on a new `postgres` service via `depends_on: condition: service_healthy`.
- `app` no longer needs the `./data` SQLite volume; Postgres has its own `./data/pg` volume.
- R2 bucket goes private — the `videos.<domain>` custom domain binding is removed at cutover.
- R2 CORS gains `GET` from `*` (signed-GET URLs are cross-origin to `record.<domain>`).
- AWS SDK presigner is used for `GetObjectCommand` in addition to `PutObjectCommand`.

## 5. Data model

Schema is defined in `app/src/db/schema.ts` using Drizzle; migration files live under `app/db/migrations/` and are applied at app boot via `drizzle-orm/node-postgres/migrator`.

The equivalent SQL:

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;  -- for gen_random_uuid()

CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         TEXT NOT NULL UNIQUE,         -- normalized lowercase
  display_name  TEXT NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE sessions (
  token_hash    BYTEA PRIMARY KEY,            -- SHA-256 of cookie token (raw bytes)
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  expires_at    TIMESTAMPTZ NOT NULL,         -- created_at + 30d (absolute)
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_sessions_user_id ON sessions(user_id);
CREATE INDEX idx_sessions_expires_at ON sessions(expires_at);

CREATE TABLE signin_tokens (
  token_hash    BYTEA PRIMARY KEY,            -- SHA-256 of magic-link token (raw bytes)
  email         TEXT NOT NULL,                -- lowercase
  expires_at    TIMESTAMPTZ NOT NULL,         -- created_at + 15m
  consumed_at   TIMESTAMPTZ,                  -- non-null = used (single-use)
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_signin_tokens_expires_at ON signin_tokens(expires_at);

CREATE TABLE recordings (
  slug          TEXT PRIMARY KEY,             -- unchanged contract from v0.1
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  r2_key        TEXT NOT NULL,
  mime_type     TEXT NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_recordings_user_id_created_at ON recordings(user_id, created_at DESC);
```

### Choices and why

- **`TIMESTAMPTZ` throughout**, replacing v0.1's `unix epoch ms`. Postgres-native; easier to inspect in `psql`. The Recording module converts at the edge where it matters.
- **UUIDs for `users.id`** via `gen_random_uuid()` (UUID v4). Avoids leaking user counts. UUID v7 (time-sortable) would be nicer for index locality but needs an extension or app-side generation; v4 is fine at v0.4 scale.
- **`BYTEA` for both token hashes.** SHA-256 of the raw token, stored as raw bytes — no hex/base64 overhead. Even if Postgres is exfiltrated, magic-link tokens and session cookies are not recoverable.
- **`ON DELETE CASCADE`** on `sessions.user_id` and `recordings.user_id`. Account-deletion isn't in v0.4 scope, but the schema is ready when it lands.
- **`recordings.user_id` is `NOT NULL`.** Every recording has an owner — no anonymous orphans. Matches the "accounts required to record" decision.
- **Absolute 30-day session expiry, no sliding refresh.** Re-auth every 30 days. Simple to reason about; bump later if friction warrants it.

### Periodic cleanup

An in-process hourly timer runs:

```sql
DELETE FROM sessions       WHERE expires_at < NOW() - interval '7 days';
DELETE FROM signin_tokens  WHERE expires_at < NOW() - interval '7 days';
```

The grace window past `expires_at` is just to avoid surprising a user whose clock skewed at the moment of expiry. Cheap and self-contained — no separate cron.

## 6. API surface

| Method | Path                       | Auth         | Purpose |
| ------ | -------------------------- | ------------ | ------- |
| GET    | `/`                        | session req. | Recorder page; 302 to `/signin` if no session |
| GET    | `/signin`                  | none         | Sign-in page (static HTML form, single `email` field) |
| POST   | `/auth/request-link`       | none         | Body `{email}` → mint signin token, send Resend email. Always returns `204`. |
| GET    | `/auth/callback?token=...` | none         | Validate token, upsert user, create session, set cookie, 302 to `/`. On failure: render "invalid or expired link" page with "request a new link" CTA. |
| POST   | `/auth/signout`            | session req. | Delete session row, clear cookie, 302 to `/signin` |
| GET    | `/me`                      | session req. | JSON `{id, email, displayName}` — used by recorder/library JS for header chrome |
| POST   | `/create-upload`           | session req. | Unchanged v0.1 contract; now also inserts `user_id` |
| GET    | `/library`                 | session req. | Library page HTML, with recordings list rendered server-side |
| GET    | `/api/recordings`          | session req. | JSON list of caller's recordings (`?limit=50&before=<created_at>` cursor); used for client-side pagination |
| DELETE | `/recordings/:slug`        | session req. | Ownership-checked. Deletes R2 object then Postgres row. |
| GET    | `/v/:slug`                 | none         | Viewer page; mints signed-GET, embeds in `<video src>` |
| GET    | `/healthz`                 | none         | `SELECT 1` against Postgres; 200/503 |

### Auth middleware contract

A single `requireSession` middleware reads the `mool_session` cookie, hashes it, looks up `sessions.token_hash`, verifies `expires_at > NOW()`, and either attaches `req.user = {id, email, displayName}` (joined from `users`) or rejects:

- **HTML routes** → 302 to `/signin`.
- **JSON / DELETE routes** → 401 with `{error: "unauthenticated"}`.

Routes are tagged at registration as `html` or `json` so the middleware knows which rejection mode to use.

### Account-enumeration posture

`POST /auth/request-link` returns `204` for any syntactically valid email, whether or not the user exists. The actual email send happens regardless — a new user is created on first callback. There is no "this email is not registered" leakage in any response.

### CORS

All app routes are same-origin. No CORS changes on the app. CORS on the R2 bucket is updated separately (see §9).

## 7. Auth and sessions

### Magic-link token

- 32 random bytes from `crypto.randomBytes`, encoded as base64url (43 chars). This is what appears in the email URL.
- The DB stores `sha256(token)` as `BYTEA`. The raw token is never persisted.
- Expiry: 15 minutes from issuance.
- Single-use: marked `consumed_at = NOW()` on first redemption. Already-consumed tokens render the same "invalid or expired" page as expired ones.
- **One pending token per email at a time:** when `POST /auth/request-link` fires, delete any unconsumed `signin_tokens` rows for that email before inserting the new one. Prevents an attacker who has gained brief inbox access from accumulating a pool of usable tokens, and ensures the latest request is the one that works.

### Session token

- 32 random bytes, base64url. The DB stores `sha256(token)` as `BYTEA`. The raw value lives only in the cookie.
- Cookie attributes:
  ```
  mool_session=<raw>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000
  ```
- **`SameSite=Lax`** (not Strict): needed because the magic-link click originates as a cross-site GET from the email client. Strict would block the cookie on that first redirect.
- **`Secure`** is env-gated (`COOKIE_SECURE=false`) for local dev over HTTP. Production always `Secure`.
- 30-day absolute expiry, no sliding refresh.

### Rate limiting

Not in v0.4. Cloudflare edge defaults are the only floor. The Resend free tier (100/day, 3k/mo) is a soft ceiling against runaway abuse on `POST /auth/request-link`. Per-IP token-bucket rate limiting is deferred to v0.5.

## 8. Viewer access (signed-GET)

### R2 bucket flip

- The `videos.<domain>` Cloudflare custom domain binding is **removed** at cutover. The bucket has no public surface afterward — all reads go through signed-GET URLs.
- CORS on the bucket:
  - `PUT` from `https://record.<domain>` (unchanged from v0.1, for uploads)
  - `GET` from `*` with no required headers (so signed-GET URLs work in the viewer's browser, which is cross-origin to R2)

### Minting signed-GET URLs

- Same AWS SDK v3 presigner already used for `PutObjectCommand`, now also used for `GetObjectCommand`.
- TTL: **1 hour** by default. Configurable via env (`VIEW_URL_TTL_SECONDS=3600`) so the knob exists without code change.
- URL form: `https://<account>.r2.cloudflarestorage.com/<bucket>/<slug>.webm?X-Amz-Signature=...`.
- Server mints fresh per page load. No caching — the presigner is in-process HMAC, cheap.

### TTL behavior

- 1h covers realistic single-session viewing in one shot.
- If the page is left open and playback starts after >1h, the next byte fetch 403s and HTML5 video errors. **Recovery: full page reload reissues a fresh URL.** Auto-refresh on 403 is deferred to v0.5.
- The signed URL is bound to bytes, not to a viewer. Sharing the signed URL would let a third party stream for an hour — same threat model as a leaked Loom URL within its session. The canonical share link (`record.<domain>/v/<slug>`) remains the stable, infinitely-shareable form.

### Edge caching

Cloudflare may cache the signed-GET response by URL. Since each signed URL is unique (signature varies per mint), there is no cross-viewer cache hit out of the box. Acceptable for v0.4 — same effective byte-path as v0.1. Cache-key normalization (stripping the signature) is a v0.5+ concern, only if egress costs ever matter.

## 9. Deployment changes

### `docker-compose.yml` gains a third service

```yaml
services:
  app:
    # unchanged shape; new env vars added below
    depends_on:
      postgres:
        condition: service_healthy
    # no longer needs the ./data SQLite volume

  cloudflared:
    # unchanged

  postgres:
    image: postgres:17-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: mool
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: mool
    volumes:
      - ./data/pg:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U mool -d mool"]
      interval: 5s
      timeout: 5s
      retries: 10
    # No published ports — only `app` connects, over the compose network
```

### New env vars (in `.env`)

```
# Postgres
POSTGRES_PASSWORD=...
DATABASE_URL=postgres://mool:${POSTGRES_PASSWORD}@postgres:5432/mool

# Resend
RESEND_API_KEY=re_...
RESEND_FROM=auth@<your-domain>           # verified sender domain in Resend

# Sessions and tokens
SIGNIN_TOKEN_TTL_SECONDS=900             # 15 min
SESSION_TTL_SECONDS=2592000              # 30 days
VIEW_URL_TTL_SECONDS=3600                # 1 hour

# Cookie
COOKIE_SECURE=true                       # set false in local-dev HTTP
```

### One-time Resend setup (manual, outside this repo)

1. Create a Resend account and an API key.
2. Add `<your-domain>` as a sender domain in Resend.
3. Add the SPF, DKIM, and DMARC DNS records that Resend prescribes (TXT/CNAME records on `<your-domain>`).
4. Wait for the domain to show "Verified" in Resend before the first production send.
5. Save `RESEND_API_KEY` and `RESEND_FROM` to the server's `.env`.

### One-time R2 changes (manual)

1. Detach the `videos.<domain>` custom domain from the bucket in the Cloudflare R2 dashboard.
2. Update the CORS policy: keep `PUT` from `https://record.<domain>`, add `GET` from `*`.

### Drizzle migrations

- Schema in `app/src/db/schema.ts`. Migration files generated via `drizzle-kit generate` into `app/db/migrations/`.
- Applied at app boot via `drizzle-orm/node-postgres/migrator` (`migrate(db, { migrationsFolder: './db/migrations' })`).
- Single-instance and migrate-on-boot are sufficient for v0.4. If multi-instance ever lands, switch to a one-shot `migrate` compose service to avoid races.

### Backups

A nightly host-side cron: `pg_dump | gzip > backups/mool-$(date +%F).sql.gz`, with 7-day retention. Operationally adjacent to v0.4 — strongly recommended at cutover since user data now lives only in Postgres.

## 10. Cutover from v0.1

Per the brainstorming decision: **fresh start, no SQLite → Postgres migration; R2 bucket wiped.**

### Cutover steps (in order, single maintenance window)

1. **Build and push the v0.4 image** (`ghcr.io/toyinogun/mool:v0.4.0`). Don't bring it up yet.
2. **Stop v0.1:** `docker compose down`.
3. **Archive the old SQLite file:** `mv data/db.sqlite data/db.sqlite.v0.1.bak`. Costs nothing; useful safety net.
4. **R2 surgery in the Cloudflare dashboard:**
   - Detach the `videos.<domain>` custom-domain binding.
   - Update CORS: add `GET` from `*` alongside the existing `PUT` from `https://record.<domain>`.
5. **Wipe the R2 bucket** of v0.1 objects. Without a Postgres row, they cannot be served anyway; this avoids paying for orphan storage. Old share links die — acceptable per the fresh-start decision.
6. **Resend setup:** add `<your-domain>` as a sender, install SPF/DKIM/DMARC DNS records, wait for "Verified". Save API key.
7. **Update `.env`** with the new vars from §9.
8. **`docker compose up -d`.** `postgres` starts and becomes healthy; `app` runs Drizzle `migrate()` on boot, creating the four tables; `cloudflared` reconnects.
9. **Smoke test** (incognito where noted):
   - Visit `record.<domain>` → expect redirect to `/signin`.
   - Enter your email → expect a Resend email within a few seconds.
   - Click the link → expect to land at `/` signed in, with header chrome populated from `/me`.
   - Record a 5-second clip → confirm upload succeeds and the slug is returned.
   - Open the share link in an incognito window → expect the viewer to play via the signed-GET URL.
   - Open `/library` → expect to see the new recording with a working delete button.
   - Delete it → confirm both the R2 object and the Postgres row are gone.
10. **Sign out and sign in again** to confirm session round-trip is clean.
11. **Install the `pg_dump` backup cron** on the host (see §9 — Backups). User data now lives only in Postgres; the first backup should run before the maintenance window closes.

### Rollback plan (if cutover fails)

- `docker compose down`
- `mv data/db.sqlite.v0.1.bak data/db.sqlite`
- Re-attach the `videos.<domain>` custom domain in Cloudflare R2
- Revert CORS to v0.1 shape
- Deploy the previous v0.3 image
- Total time: ~5 minutes if the v0.3 image tag is still in GHCR.

### What stops working at cutover

- Any old `record.<domain>/v/<slug>` share link → 404.
- Any direct `videos.<domain>/<slug>.webm` URL → DNS resolves but bucket binding is gone → 404.

### What starts working

- Sign-up and sign-in via magic link.
- The library page.
- Private-bytes viewing via signed-GET.

## 11. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation in v0.4 |
| --- | --- | --- | --- |
| Resend deliverability — magic links to spam (Gmail/Outlook) | Medium | Sign-up flow blocked | DKIM + SPF + DMARC correctly installed; pre-cutover test sign-in to both a Gmail and an Outlook address |
| Postgres single-instance — container or volume corruption | Low | Total user-data loss | `pg_dump` nightly cron at cutover; `./data/pg` on the host FS, not in the container layer |
| R2 CORS misconfig — signed-GET 403 in viewer browsers | Medium | Viewer page cannot play bytes | Incognito smoke-test in step 9 of cutover catches this |
| `SameSite=Lax` (not Strict) | Low | Light CSRF surface on authed endpoints | Lax blocks cross-site POST/DELETE by default; the surface is small (signout, delete recording). Revisit if a real attack emerges. |
| Local-dev cookies need `Secure=false` | Low | Dev confusion if forgotten | Env-gated `COOKIE_SECURE` |
| Magic-link delay → user impatience → multiple requests | Low | UX confusion | "Request a new link" CTA on the sign-in page after first request; only the newest token is valid (§7) |
| Resend free-tier exhaustion (100/day, 3k/mo) from spam | Low | Sign-up flow temporarily blocked | Cloudflare edge bot-fight is the v0.4 floor; per-IP rate limiting deferred to v0.5 |

## 12. Open questions deferred to implementation

These are settled when the implementation plan is written:

- Exact Drizzle schema TS shape (column names, type imports, relation helpers).
- Express middleware ordering (helmet, compression, cookie-parser, body parsers, `requireSession`).

- `/library` HTML — likely server-rendered via a small template helper since the recordings list is per-user; folded into the same Express static + handler setup as v0.1's recorder/viewer.

## 13. Growth-ladder pointer

The v0.1 spec §12 named v0.4 as the rung where two structural changes happen: SQLite → Postgres, and public bucket → presigned-GET access. This spec executes both, alongside the accounts work that ties them together. Subsequent rungs (v0.5 polish/thumbnails, v0.6 comments) build on a stable foundation: every recording has an owner, bytes are private behind signed URLs, and the metadata store can grow without re-architecting.
