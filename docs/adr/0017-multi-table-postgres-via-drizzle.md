# Multi-table Postgres via Drizzle ORM

v0.4 replaces the single-table SQLite store with a four-table Postgres schema managed through Drizzle ORM.

## Why

v0.3 stored one table (`recordings`) in a SQLite file. That was right-sized for an anonymous, single-instance recorder: one writer, one table, no user concept. v0.4 adds authentication (users, sessions, signin tokens) and ownership on recordings — four tables, foreign keys, server-managed UUIDs, and a requirement that the data survive container restarts in a healthy, queryable state. SQLite carries all of that technically, but the ops story diverges: Postgres ships as a first-class Docker image with a built-in healthcheck, production tooling (`pg_dump`), and the `pg_isready` probe the compose `depends_on` needs.

Toyin's stated preference is also Postgres for anything beyond a scratch store.

## Decision

Schema in TypeScript via Drizzle's `pgTable` helpers (`app/src/db/schema.ts`). Migrations generated by `drizzle-kit generate` into `app/db/migrations/`. Applied at app boot via `drizzle-orm/node-postgres/migrator.migrate`.

Four tables, matching §5 of the v0.4 spec exactly:

- `users` — email + display name, UUID PK (`gen_random_uuid()`).
- `sessions` — `bytea` token hash PK (SHA-256 of the raw 32-byte secret), FK to `users(id) ON DELETE CASCADE`, `expires_at`. Indexed on `user_id` and `expires_at`.
- `signin_tokens` — same `bytea` hash PK, email, `expires_at`, `consumed_at`. Indexed on `expires_at`.
- `recordings` — slug PK, FK to `users(id) ON DELETE CASCADE`, `r2_key`, `mime_type`. Indexed on `(user_id, created_at DESC)` for the library query.

## Rationale

**Schema-as-TS** gives column types for free at the call sites — no separate type files or runtime coercions needed. Generated migrations remove the class of drift where the applied schema diverges from what the code expects. The migration SQL is committed to the repo and human-readable; there are no implicit schema changes at runtime beyond what `migrate()` applies from the committed files.

**Migrate-on-boot** is race-free for the v0.4 single-instance deployment because `docker-compose.yml` declares:

```yaml
app:
  depends_on:
    postgres:
      condition: service_healthy
```

Postgres is healthy before `app` starts. `app` calls `migrate()` before `app.listen`. The listener only opens after migrations succeed, so the app never accepts traffic against a partially-migrated schema.

**Indexes:** four are added beyond the primary keys — `idx_sessions_user_id`, `idx_sessions_expires_at`, `idx_signin_tokens_expires_at`, `idx_recordings_user_id_created_at`. The first covers the session-lookup join. The second and third support the nightly cleanup loop that hard-deletes expired rows. The fourth covers the library `SELECT … WHERE user_id = $1 ORDER BY created_at DESC` query.

## Consequences

- The full schema lives in `app/src/db/schema.ts`; the generated SQL is in `app/db/migrations/`.
- When multi-instance deployment lands (Mool grows), migrate-on-boot becomes a race — two `app` replicas may run `migrate()` concurrently. Switch to a one-shot `migrate` compose service (a separate container that runs `migrate` and exits before `app` starts) at that point. This is called out in §9 of the v0.4 spec.
- The SQLite dependency (`better-sqlite3`) is dropped entirely. No migration from SQLite data is performed — v0.4 is a fresh start (see spec §10).
- Tests do not need a Postgres instance. The test layer uses in-memory fakes (see ADR-0018). `npm test` runs without Docker.

Reference: v0.4 spec §5 (data model) and §9 (deployment changes).
